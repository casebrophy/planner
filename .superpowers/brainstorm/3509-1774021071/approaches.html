Implementation Approach

Two parallel tracks — how should we structure the work?

A

Max Parallelism (6 agents)

Split backend into 5 independent worktree agents + 1 frontend agent. Each backend agent handles one activation piece. Frontend builds clarification queue view.

Pros

  • Fastest wall-clock time
  • Each agent has a small, focused scope
  • True isolation via git worktrees

Cons

  • 6 concurrent agents = high token cost
  • Merge conflicts likely between backend agents touching shared files (main.go, mcpapp)
  • Hard to review 6 branches at once
B

Two-Track Parallel (2 agents) — Recommended

One frontend agent builds the clarification queue view + polish. One backend agent works through all 6 activation pieces sequentially in a structured order. Both run in parallel worktrees.

Pros

  • No merge conflicts within each track
  • Backend agent handles shared files (main.go, mcpapp) coherently
  • Easy to review: 2 branches, 2 diffs
  • Still gets parallelism where it matters (FE ∥ BE)

Cons

  • Backend agent has a large sequential scope
  • If one backend piece blocks, it stalls the rest
C

Three-Track Parallel (3 agents)

Frontend agent + Backend-core agent (dispatcher, ingestion wiring, debrief, jobs) + Backend-MCP agent (all 5 new MCP tools). MCP tools are fully independent from activation logic.

Pros

  • Good parallelism without conflict risk
  • MCP tools are cleanly isolated in mcpapp/
  • 3 branches — manageable review

Cons

  • MCP agent may finish fast and sit idle
  • Slight coordination needed if MCP tools need new bus methods
